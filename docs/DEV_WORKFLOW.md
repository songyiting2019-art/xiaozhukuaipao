# 开发流程

本文件用于减少重复解释和临时排错。后续线程开始任务前，应优先查看相关规则文件。

## 固定流程

## 微信小游戏方向

当前项目先用网页原型快速验证玩法和美术，但后续目标是迁移到微信小游戏。开发时默认按微信小游戏方向做判断：

- 先考虑手机竖屏、单手触屏、短局体验。
- 图片、音频和首屏资源要控制大小，避免手机加载慢。
- 动画要顺滑但克制，避免过多 DOM 或过重效果。
- 玩家关键操作不能依赖 hover、键盘或桌面鼠标习惯。
- 新增素材时要命名清楚、目录清楚，后续便于迁移。
- 新增功能时尽量让玩法逻辑和表现层分开，方便以后换成小游戏运行环境。

## 工具和流程优先原则

本项目的目标不只是完成某个小任务，也包括建立稳定、可复用的 Codex 工作方式。遇到工具、权限、预览、推送、部署、打包等问题时，优先修正长期工作路径，不要连续尝试临时绕路。

处理原则：

1. 第一次失败后先分类
   - 权限问题。
   - 工具未安装。
   - 配置不对。
   - 端口/服务管理混乱。
   - 网络或远端问题。
   - 代码本身问题。

2. 如果是工具链问题，先建设正确路径
   - 需要授权就明确告诉用户授权什么。
   - 需要安装或配置就先完成配置。
   - 需要固定流程就写进文档。
   - 不要为了眼前任务连续换三四种临时方案。

3. 只有在短期验证或明确临时场景下，才使用替代方案
   - 使用替代方案时，必须说明它是临时方案。
   - 临时方案不能变成默认流程。

4. 成功后沉淀
   - 把稳定方案写入工作流文档。
   - 同步给相关线程。

示例：GitHub 推送曾经因为 HTTPS token 反复失败，最终正确方案是配置 SSH key，并将仓库远端固定为 SSH。以后不再回到 token 绕路。

## 线程同步原则

文档是主渠道，线程消息只是任务触发器。更新文档后，不默认群发所有线程，避免每个线程都进入一轮完整思考。

同步规则：

- 规则类文档更新：默认不主动通知所有线程。
- 某线程正在执行相关任务：发送短提醒。
- 即将给某线程布置任务：在任务卡中引用相关文档。
- 紧急影响当前工作的规则变化：只通知受影响线程。

同步消息应尽量短，例如：

```text
DEV_WORKFLOW.md 已更新工具流程规则。下次任务前请读取。
```

### 1. 开始前

- 先查看当前工作区状态，确认是否有其他线程或用户留下的改动。
- 不覆盖与本任务无关的改动。
- 明确本次修改范围和禁止范围。

### 工作区已有改动时

发现工作区有未提交改动时，不要只说“绕开它们”。先把工作区变成可解释状态：

1. 盘点改动
   - 查看哪些文件被修改、删除或新增。
   - 区分主游戏、关卡编辑器、UI素材、文档、生成包、临时文件。

2. 判断归属
   - 本轮新改动。
   - 用户手动改动。
   - 其他线程正在处理的改动。
   - 历史残留或不确定来源。

3. 判断关系
   - 和本次任务无关：隔离处理，不混入提交。
   - 和本次任务相关：先理解内容，再继续。
   - 可能冲突：暂停说明风险，必要时请总控或用户决定。

4. 选择处理方式
   - 单独提交当前任务。
   - 等相关线程交回后统一收口。
   - 建分支或临时备份。
   - 清理明确的临时文件。
   - 对不确定改动，不擅自删除或回退。

最终目标：每个未提交改动都应知道“属于什么、为什么存在、下一步谁处理”。

### 2. 修改中

- 小改动直接在对应文件中完成。
- 大改动先拆阶段，不一次性重构太多。
- 视觉实验先进预览页，不直接污染正式 `index.html`。
- 关卡编辑器改动只放在 `level-editor.*`。

### 素材管理

根目录只放项目入口、代码、样式和文档，不放生图、参考图、临时图、视频或压缩包。

素材统一放在 `assets/` 下：

- `assets/optimized/`：正式页面正在引用的压缩图。
- `assets/audio/`：正式页面正在引用的压缩音频。
- 候选图、候选音频、原始 wav、生图初稿、旧归档素材和临时预览文件不再提交入库。

正式代码只引用 `assets/optimized/` 和 `assets/audio/` 根目录里的素材。其他目录里的素材默认不是线上资源，任务结束前清理或不要提交。

### 3. 检查

常用检查：

```bash
node --check game.js
git diff --check
```

如有新增 JS 文件，应一起做语法检查。

### 任务结束检查清单

任务完成前，快速确认：

- 是否改动了正确范围。
- 是否误碰其他线程或用户改动。
- 是否运行必要检查。
- 是否需要预览。
- 是否需要更新文档。
- 是否需要通知相关线程。
- 是否需要提交、推送、打包。
- 是否留下临时文件或未解释的工作区改动。

### 4. 本地预览

优先使用本地服务预览。多线程并行时，不共用同一个端口，每个线程固定自己的端口。

普通静态预览：

```bash
python3 -m http.server <端口号> --bind 127.0.0.1
```

关卡编辑器预览和本机部署必须使用专用服务：

```bash
python3 tools/level_editor_server.py --port 8033
```

固定端口池：

```text
总控线程：8030
玩法代码线程：8031
UI背景线程：8032
关卡编辑器线程：8033
音效声音线程：8034
```

常用地址：

```text
总控玩家页：http://127.0.0.1:8030/index.html
玩法玩家页：http://127.0.0.1:8031/index.html
UI预览页：http://127.0.0.1:8032/index.html
关卡编辑器：http://127.0.0.1:8033/level-editor.html
关卡编辑器试玩页：http://127.0.0.1:8033/index.html
音效线程已归档，当前仓库不再保留音效候选预览页。
```

预览端口规则：

- 不要所有线程抢 `8030`。
- 不要随意跳到新端口。
- 如果本线程固定端口不可用，先查清原因，而不是直接连续换端口。
- 关卡编辑器点击“试玩”会写入浏览器本机试玩数据，并打开同端口游戏页；编辑器在 `8033`，试玩也用 `8033/index.html`。
- 关卡编辑器点击“部署正式”会通过专用本地服务写回 `levels.js`；如果不是用 `tools/level_editor_server.py` 启动，部署不会生效。
- 确实需要换端口时，说明原因和最终使用地址。

### 5. 推送

仓库已配置 SSH 推送：

```text
git@github.com:songyiting2019-art/xiaozhukuaipao.git
```

当前仓库固定使用：

```text
~/.ssh/id_ed25519_github
```

后续推送应直接走：

```bash
git push origin main
```

不要再回到 HTTPS token 流程。

### 6. 打包

需要发给外部查看时，用当前确认版本打包：

```bash
git archive --format=zip --output xiaozhukuaipao-v1-<commit>.zip HEAD
```

## 输出要求

最终回复优先说明：

- 改了什么。
- 验证了什么。
- 是否推送。
- 是否有未处理风险。

不要默认贴大段 diff，除非用户要求。
