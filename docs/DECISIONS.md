# 决策记录

记录已经确认的重要项目决策，避免反复讨论。

## 协作方式

- 采用“总控收口 + 专项线程执行”。
- 简单明确的专项任务可以直接找对应线程。
- 复杂、跨模块、不确定、发布相关任务找总控。
- 最终提交、推送、打包、版本同步由总控收口。

## 产品方向

- 当前先做网页原型，后续目标是微信小游戏。
- 玩法、UI、素材、音效、预览和工具链，都默认按手机竖屏小游戏方向判断。
- 不为了网页方便牺牲后续小游戏迁移，例如把动态数据画进图片、依赖 hover、使用过重素材。

## GitHub 推送

- 使用 SSH 推送。
- 远端：`git@github.com:songyiting2019-art/xiaozhukuaipao.git`
- 不再默认使用 HTTPS token 方式。

## UI 第二版

- 图片负责质感，CSS 负责结构、状态和适配，JS 负责数据和交互逻辑。
- `美术UI二版｜视觉资产` 负责生图和接图。
- 用户不需要判断用生图、CSS 还是 JS，由 UI线程按规则判断。
- 生图资产先预览，确认后再接入正式游戏。
- 暂不单独开生图线程。

## 第二版棋盘规格

- 正式棋盘已从 `12×19` 切换为 `10×16`。
- 四角圆弧保留，外轮廓视觉大小基本不变，格子和小猪比第一版更大。
- 正式关卡重做为 15 关，从 12 只逐步增加到 45 只。
- 旧 `12×19` 关卡不再作为正式关卡使用。
- 棋盘、进度、收藏解锁和编辑器试玩存储 key 已升级，避免旧进度污染新规格。
- 后续 UI 背景、棋盘外框和关卡编辑器都默认按 `10×16` 设计。

## 关卡编辑器

- 关卡编辑器是开发者工具，不是玩家页面。
- 独立页面 `level-editor.html`。
- 不放进玩家首页入口。
- 本机开发采用“专用本地服务启动，试玩，确认后自己点击部署正式写入 `levels.js`，再由总控推 GitHub”的流程。
- 远程无法本机部署时，才使用“导出给总控”。
- 暂不做编辑器直接写仓库或直接发布正式关卡。

## 代码拆分

- 不一次性大拆 `game.js`。
- 第一阶段：拆 `levels.js`。
- 第二阶段：拆 `board-core.js`，让主游戏和关卡编辑器共用棋盘、占格和基础校验规则。
- 后续考虑：`scoring.js`。
- 道具、收藏馆、UI 模块等高变化部分后拆。

## 预览

- 多线程预览使用固定端口池：总控 `8030`，玩法 `8031`，UI `8032`，关卡编辑器 `8033`。
- 不随意开新端口；固定端口不可用时先查原因。
- 关卡编辑器试玩时，编辑器页和游戏页必须使用同一个端口。
- UI 和视觉实验先放预览，不直接污染正式页面。
