# UI 工作流

第二版 UI 的目标是提升整体质感，不再用 CSS 硬画复杂美术。UI背景线程负责生图和接图，但必须先做资产预览，确认后再接入正式游戏。

后续目标是微信小游戏，所以 UI 默认按手机竖屏小游戏设计。画面要优先服务中间核心游戏区，背景和装饰要增强质感，但不能抢操作空间，也不能造成加载过慢。

## 工具选择规则

用户不需要判断“这里用生图还是 CSS”。UI 线程应主动判断。

### 优先用生图的情况

目标是高级质感、自然材质、插画氛围、游戏美术感时，优先生图。

典型内容：

- 首页主视觉、背景氛围图。
- 草地、土路、木牌、栅栏、门、小屋、花草装饰。
- 棋盘外框、自然边缘、泥土和草皮质感。
- 按钮底图的材质感。
- 弹窗底板、收藏馆卡片底板。
- 道具图标、动物素材、皮肤展示图。
- 解锁弹窗中的精致展示物。

要求：

- 不把动态文字和玩家数据画进图片。
- 星星数、关卡数、按钮文案、已携带、未解锁等由代码叠加。
- 注意手机竖屏比例、安全区、留白和后续接入。
- 控制素材尺寸和数量，避免首屏加载过慢。
- 先进入预览页，不直接改正式页面。

### 优先用 CSS 的情况

目标是布局、位置、响应式、状态变化、文字排版、点击反馈时，优先 CSS。

典型内容：

- 元素摆放、间距、对齐。
- 手机适配和缩放。
- 文字字号、颜色、行高、数字位置。
- 未解锁遮罩、可解锁高亮、已携带标签、选中状态。
- 按钮按下、轻微跳动、发光、弹窗出现、提示脉冲。
- 网格线、透明叠层、遮罩。

### 需要 JS/玩法配合的情况

目标依赖玩家数据、游戏状态、解锁状态、道具次数、关卡进度时，需要 JS 状态配合。

典型内容：

- 星星数、已通关数实时变化。
- 道具是否解锁、是否已携带。
- 炮仗、翻转、兴奋剂是否显示。
- 结算页是否提示有新内容可领取。
- 收藏馆点击后显示不同说明。
- 玩家选择皮肤后动物改变。

涉及玩法规则或新增状态字段时，先交回总控拆分。

## 判断口诀

- 要画得好：用生图。
- 要摆得准：用 CSS。
- 要跟玩家数据变化：用 JS 状态。
- 要能点击、能切换、能禁用：CSS + JS。
- 要自然纹理、插画感、精致图标：生图。
- 文字、数字、标签不要画进图里。

## 第二版 UI 优先方向

- 首页主视觉。
- 游戏背景 / 棋盘外框。
- 主按钮底图。
- 通关弹窗底板。
- 收藏馆卡片质感。
- 道具图标统一风格。

## 素材目录规则

UI 线程不要把生图、参考图、临时图直接放到项目根目录。所有素材必须进入 `assets/` 下的对应目录。

建议结构：

```text
assets/
  optimized/          正式游戏正在使用的压缩图
```

规则：

- 正式页面只引用 `assets/optimized/` 里的图片。
- 生图初稿、原始参考图、临时对比图和旧版本素材只做本地临时文件，不提交入库。
- 用户确认采用后，再裁剪、压缩、命名清楚，并只把正式版本放进 `assets/optimized/`。
- 根目录不放图片、视频、压缩包或临时预览产物。

## 流程

1. 收到 UI 需求后，先判断它属于生图资产、CSS布局、JS状态，还是混合任务。
2. 生图资产先做 2-3 个预览方案，不直接改正式游戏页。
3. CSS布局优先在预览页或局部页面验证。
4. 涉及 JS 状态或玩法规则，先交回总控拆分。
5. 用户确认视觉方向后，再接入正式游戏。
