# 项目协作规则

本项目采用“总控收口 + 专项线程执行”的方式。用户不需要判断任务该由谁做；能明确归属的专项任务可以直接找专项线程，复杂、跨模块、不确定、发布相关的任务回到总控。

## 产品方向

本项目当前是网页原型，但后续目标是做成微信小游戏。因此所有玩法、UI、素材、音效和工具链决策，都要按手机竖屏小游戏来考虑：

- 优先保证触屏操作清楚、点击区域可靠、节奏适合手机短时间游玩。
- 资源要轻量，避免大图、大音频和过多首屏加载。
- 视觉素材、音效素材和代码结构要方便后续迁移到微信小游戏环境。
- 不依赖桌面浏览器特有的交互方式，例如 hover、键盘快捷键、超大屏布局。
- UI 不把玩家数据画死在图片里，星星、关卡、道具状态等必须由代码叠加。
- 任何新功能都要考虑手机性能、加载速度、弱网和重复游玩体验。

## 线程分工

### 总控线程

负责需求拆分、方案判断、跨线程协调、最终检查、提交、推送、打包和版本同步。

适合找总控的任务：

- 新道具、新系统、新动物这类跨玩法/UI/素材的功能。
- 不确定属于玩法、UI还是关卡的问题。
- 需要形成版本、推 GitHub、打包、同步各线程。
- 工具链、工作流、项目结构调整。

### 玩法代码线程

负责规则和逻辑：

- 移动、阻挡、撞击、出场。
- 道具判定、得分、星级、解锁条件。
- 存档、收藏馆状态、动物占格、关卡规则。
- 无解检测、关卡难度逻辑建议。

边界：

- 不主动做首页视觉、背景图、按钮美术、动物素材精修。
- 不主动改 `level-editor.*`。
- 不负责最终推送和打包。

### UI背景线程

负责玩家可见表现：

- 首页、游戏页、收藏馆、结算页视觉。
- 背景、生图素材、按钮、提示、弹窗、动效观感。
- 动物外观、道具图标、手机适配。
- 视觉预览页。

边界：

- 不主动改玩法规则、道具判定、得分、星级、关卡数据。
- 需要新增状态字段或玩法配合时，交回总控。
- 不负责最终推送和打包。

### 音效声音线程

负责声音体验：

- 按钮点击、小猪移动、撞击、进门、炮仗、翻转、兴奋剂、解锁、结算等音效。
- 音量层级、触发时机、是否打扰玩家、手机外放听感。
- 音效预览页和候选素材整理。

边界：

- 不主动改玩法规则、关卡数据、UI 布局和美术风格。
- 需要游戏事件触发点时，交回总控或玩法代码线程接入。
- 不负责最终推送和打包。

### 关卡编辑器线程

负责开发者工具：

- `level-editor.html`
- `level-editor.css`
- `level-editor.js`
- 关卡摆放、编辑、检查、导出。

边界：

- 不把编辑入口放进正式玩家页面。
- 不主动改正式玩法规则和玩家页视觉。
- 不直接发布正式关卡，不负责推送。

## 工作原则

- 简单明确的专项任务，可以直接找专项线程。
- 复杂混合任务，先找总控。
- 多线程工作时，不允许多个线程同时大改同一文件。
- 子线程发现跨界问题时，先报告或请求总控拆分，不擅自越界大改。
- 最终发布、打包、同步，由总控收口。
